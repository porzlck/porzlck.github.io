CS:APP 第五章

发表于2026-04-24|更新于2026-04-24|CS:APP
|浏览量:

本章主线

第五章的核心问题是：在程序结果正确的前提下，怎样让程序运行得更快。

这一章最重要的转变是：

  • 正确性关注“能不能做出来”
  • 性能关注“要花多大代价做出来”

也就是说，两段代码可能都能得到同样的结果，但执行代价完全不同。

5.1 为什么程序会慢

程序慢，通常不只是因为“电脑不够好”，更常见的原因是：

  • 做了重复计算
  • 循环里有不必要的工作
  • 访问内存太频繁
  • 指令之间依赖太强，不能并行推进
  • 代码结构不利于编译器优化

所以优化的核心不是“神秘技巧”，而是：

让机器做更少的无效工作。

5.2 编译器优化的能力和局限

现代编译器确实会帮程序员做很多优化，但它不是万能的。

编译器通常擅长：

  • 消除一些简单的冗余
  • 进行常量传播
  • 改写局部表达式
  • 利用寄存器保存中间结果

但编译器也有明显局限：

  • 它不能总是确认两个指针是否会指向同一块内存
  • 它不一定敢随便改变代码顺序
  • 它必须保证程序语义不被破坏

所以本章一个重要结论是：

程序员仍然需要写出更容易被优化的代码。

5.3 如何衡量程序性能

第五章里一个非常重要的指标是 CPE。

CPE

CPE 表示：

每个元素平均花多少个周期来处理。

它的好处是：

  • 可以把不同规模的数据统一到一个可比较的指标上
  • 便于看出优化前后每个元素的平均代价是否下降

你可以先粗暴理解成：

CPE 越小，程序越快。

5.4 优化的第一原则：先找主要瓶颈

并不是所有代码都值得优化。

真正值得优化的，通常是：

  • 高频执行的循环
  • 大规模数据处理路径
  • 调用次数特别多的核心函数

这也呼应第一章里讲过的 Amdahl 定律：

整体性能提升，取决于你有没有优化到真正耗时的部分。

5.5 循环是性能重点

第五章为什么总盯着循环看？

因为很多程序的主要时间都花在循环里：

  • 数组遍历
  • 向量求和
  • 乘法累积
  • 字符串扫描
  • 矩阵和图像处理

循环一旦写得不好，低效就会被重复很多次。

所以这一章的很多优化，本质上都是：

减少每次循环迭代的成本。

5.6 消除循环中的低效操作

一个非常常见的优化思路是：

不要把每次循环都一样的工作，重复做很多遍。

例如：

  • 循环边界能提前算好，就不要每次重算
  • 不变的表达式能拿到循环外，就尽量拿出去
  • 长度、基地址、结束位置等，能缓存就缓存

这一类优化的本质是：

把循环体变小。

5.7 减少过程调用开销

函数调用很有价值，但也不是免费。

调用一个函数通常意味着：

  • 传参数
  • 保存现场
  • 跳转
  • 返回

如果某个特别简单的小操作被放在高频循环里反复调用，调用本身可能就成了负担。

因此在核心循环里，性能优化常会考虑：

  • 减少不必要的小函数调用
  • 让编译器有机会内联

这一部分的核心不是“函数不好”，而是：

在高频路径上，额外开销会被放大。

5.8 减少不必要的内存引用

这一章最值钱的性能直觉之一就是：

访问内存通常比访问寄存器贵。

所以如果一个中间结果可以暂时保存在寄存器里，就不要每一步都写回内存再读出来。

常见优化思路：

  • 把中间累计结果存在局部临时变量中
  • 循环结束后再一次性写回
  • 避免反复读写同一个内存位置

这一点可以直接记成：

少碰内存，多用寄存器。

5.9 循环展开

循环展开是第五章的一个经典优化技巧。

什么是循环展开

原来一次循环只处理一个元素，现在改成一次处理多个元素。

例如：

  • 原来每轮处理 1 个
  • 现在每轮处理 2 个、4 个，甚至更多

为什么有用

因为它可以：

  • 减少循环控制本身的开销
  • 让更多操作有机会并行推进
  • 降低每个元素平均分摊到的跳转和判断代价

但循环展开也不是无脑越大越好，因为：

  • 代码会变长
  • 寄存器压力会变大
  • 可读性会变差

5.10 提高指令级并行

第五章的另一个核心是：

不是所有操作都必须严格排成一条线。

如果多条操作之间互不依赖，处理器就有机会并行推进。

例如在求和时：

  • 如果每一步都依赖上一步的累加结果
  • 那么后面的操作就只能排队等前面完成

这会限制性能。

多个累加变量

一种常见优化方法是：

  • 不只用一个累加器
  • 而是同时维护多个部分和

这样做的好处是：

  • 减少长依赖链
  • 让更多运算可以重叠进行

这一点非常重要，可以记成：

性能瓶颈很多时候不是“算得不够快”，而是“依赖太强，没法同时算”。

5.11 结合律和重新结合

在数学里，很多运算满足结合律。
但在机器里，尤其是浮点数里，这件事要小心。

例如：

  • (a + b) + c
  • a + (b + c)

在数学上可能一样，但在浮点数里不一定完全相同。

所以优化时有一个重要边界：

  • 有些重排可以提升性能
  • 但不能为了快而随便破坏程序本来的语义

这也说明：

性能优化永远要在正确性边界内做。

5.12 性能的真正限制来自哪里

第五章并不是在教你背几个技巧，而是在教你看清性能的限制源头。

常见限制包括：

1. 延迟

  • 一条操作完成需要时间
  • 后续依赖它的操作必须等待

2. 吞吐量

  • 单位时间内能完成多少操作

3. 数据相关

  • 后一步依赖前一步结果
  • 导致不能同时推进

4. 内存访问成本

  • 数据不在寄存器里，就要付出更高代价去拿

因此第五章真正训练的是：

从“代码长什么样”切到“机器会被什么拖慢”。

5.13 本章最重要的思维模型

1. 性能优化的目标是减少平均代价

不是单纯让代码变短，而是让每个数据项的平均处理成本下降。

2. 循环是核心战场

优化最值钱的地方，通常在高频循环里。

3. 少做重复工作

循环里不变的东西，尽量拿出去。

4. 少访问内存

中间结果能保存在寄存器，就不要频繁读写内存。

5. 打断长依赖链

多个累加器、循环展开，本质上都在减少串行依赖。

6. 优化要在正确性边界内

不是所有“看起来更快”的改写都合法。

易混点整理

正确和快

  • 正确只说明结果对
  • 快还要求机器执行代价低

编译器优化和手写优化

  • 编译器很强
  • 但程序员仍然需要写出易优化的结构

循环展开和算法优化

  • 循环展开是局部执行层优化
  • 它不等于换了算法

少调用函数和函数没用

  • 不是说函数不好
  • 而是高频路径上每一点额外开销都会放大

浮点重排和数学等价

  • 数学上等价
  • 在机器里未必逐位等价

学这一章时该怎么抓重点

如果你现在不想陷进太多硬件细节，第五章至少要保住这五件事：

  1. 知道性能要靠指标来衡量，例如 CPE
  2. 知道高频循环是优化重点
  3. 知道要把循环里不必要的工作移出去
  4. 知道少访问内存、多保留中间结果
  5. 知道长依赖链会限制性能，多个累加器和展开能缓解它

一句话总结

第五章是在训练你从“程序能跑”升级到“程序跑得划算”：优化程序性能的核心，不是炫技巧，而是识别真正的瓶颈，减少重复工作、降低内存访问成本，并尽量让机器能更并行地推进计算。

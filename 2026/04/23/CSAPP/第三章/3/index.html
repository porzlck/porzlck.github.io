CS:APP 第三章

发表于2026-04-23|更新于2026-04-23|CS:APP
|浏览量:

本章主线

第三章的核心问题是：C 程序是怎样被翻译成机器级指令的，以及机器究竟是怎样通过寄存器、内存、跳转和栈来执行程序的。

如果说第二章是在回答“数据长什么样”，那么第三章就在回答“这些数据是怎么被机器真正拿来算的”。

这一章最重要的视角变化是：

  • 高级语言里看到的是变量、表达式、if、while、函数调用
  • 机器层面真正执行的是寄存器操作、访存、比较、跳转、调用和返回

3.1 机器级程序和汇编的关系

  • C 代码不会被 CPU 直接执行。
  • 编译器会把 C 程序翻译成汇编代码，再进一步变成机器指令。
  • 汇编代码是机器指令的一种可读表示形式。

本章不要把重点放在“背汇编语法”上，而要放在：

  • 机器在操作什么
  • 机器状态如何变化
  • C 语言结构如何落到机器执行路径上

一句话记忆：

汇编不是目的，汇编只是暴露机器执行方式的窗口。

3.2 机器级程序在操作什么

机器真正关心的对象主要有四类：

  • 寄存器
  • 内存
  • 立即数（常数）
  • 指令地址

寄存器

  • 寄存器是 CPU 内部的小而快的存储位置。
  • CPU 会高频地直接操作寄存器中的值。
  • 寄存器数量很少，所以只能放最常用的数据。

可以把寄存器粗暴理解成 CPU 手边的小工作台。

内存

  • 内存容量大得多，但访问速度比寄存器慢。
  • 大量变量、数组、结构体、栈帧等都要放在内存中。

立即数

  • 立即数就是指令中直接写死的常量。
  • 例如把 5 加到某个值上，5 就可以是立即数。

指令地址

  • 程序不是“一整块一起执行”的。
  • CPU 总是通过程序计数器指向下一条要执行的指令。

3.3 机器状态

第三章一个非常重要的概念是机器状态。

所谓机器状态，可以粗暴理解成这些东西当前的值：

  • 各个寄存器里的内容
  • 程序计数器当前指向哪里
  • 内存中某些位置的内容
  • 条件码的状态

所以一条指令执行的本质，就是：

根据当前状态，计算出新的状态。

例如：

  • 数据传送指令会修改寄存器或内存中的值
  • 算术指令会修改结果寄存器和条件码
  • 跳转指令会修改程序计数器
  • 调用和返回会修改栈和控制流位置

3.4 操作数和寻址

机器指令不能只操作“值”，还必须知道这个值从哪里来、要放到哪里去。

所以机器级代码总是在处理：

  • 源操作数
  • 目的操作数

常见来源有三种：

  • 某个寄存器里
  • 某个内存地址里
  • 指令自带的立即数

这一部分的核心思想是：

机器不认识变量名，只认识存储位置。

也就是说：

  • x、y 这种变量名是给人看的
  • 编译器最终要把它们映射成寄存器或内存地址

3.5 数据传送

程序执行时最常见的一件事，就是把数据从一个地方搬到另一个地方。

例如：

  • 从内存读到寄存器
  • 从寄存器写回内存
  • 把一个常数装进寄存器

这类指令本身不复杂，但极其重要，因为几乎所有计算都要先建立在“数据已经被放到合适位置”上。

一句话记忆：

计算之前，通常先搬运。

3.6 算术和逻辑运算

机器会对寄存器或内存中的值进行：

  • 加法
  • 减法
  • 与、或、异或
  • 移位

从机器角度看，这些都是对位模式做操作。

这一点和第二章是连起来的：

  • 第二章讲位模式如何表示数据
  • 第三章讲机器如何对这些位模式做运算

3.7 条件码

条件码是第三章必须抓住的一个点。

可以把条件码理解成：

CPU 对最近一次运算结果贴上的几个标签。

常见标签包括：

  • 结果是否为 0
  • 结果是否为负
  • 是否发生溢出
  • 是否产生进位

条件码的作用是：

  • 后面的跳转指令不用重新算一遍
  • 只要查看这些标签，就能决定要不要跳转

所以条件码的本质是：

把“比较结果”保留下来，交给控制流指令使用。

3.8 控制流

高级语言中的控制结构，到了机器层面都会被拆成更基本的执行路径。

if 语句

在机器里，if 本质上通常是：

  • 先比较
  • 再根据结果条件跳转

也就是：

比较 + 条件跳转

while 和 for

循环在机器层面，本质上通常是：

  • 做一次条件判断
  • 满足条件就进入循环体
  • 执行完循环体再跳回去继续判断

也就是：

比较 + 条件跳转 + 回跳

这一章最重要的转变之一，就是你要开始把代码结构看成控制流路径：

  • 顺序执行
  • 分支
  • 汇合
  • 回跳

一句话记忆：

高级语言写的是结构，机器执行的是路径。

3.9 过程调用

函数调用不是简单地“跳过去执行几行代码”。

它至少还要解决这些问题：

  • 执行完后回到哪里
  • 参数怎么传
  • 局部变量放在哪里
  • 当前函数的现场怎么保存

因此，过程调用的本质是：

控制转移 + 运行时上下文管理

为什么要保存现场

如果函数调用前的状态不保存，那么：

  • 返回地址会丢
  • 原来的寄存器值可能被覆盖
  • 局部变量会混乱

递归最能说明这一点：

  • 同一个函数代码可以被多层同时调用
  • 每一层都必须有自己独立的数据和返回位置

3.10 栈和栈帧

函数调用通常依赖栈来保存现场。

这是因为函数调用天然具有：

后进先出

例如：

  • main 调用 A
  • A 调用 B
  • 那一定是 B 先返回，再轮到 A

这和栈的规则完全一致。

栈帧

每一次函数调用，通常都会对应一个栈帧。

可以把栈帧理解成：

这一层函数调用的现场包。

里面通常会放：

  • 返回地址
  • 局部变量
  • 保存下来的寄存器值
  • 某些临时数据

递归之所以依赖栈帧，就是因为每一层调用都必须拥有自己独立的一份运行上下文。

3.11 数组、指针和地址计算

第三章还有一个非常关键的视角：

数组和指针在机器层面，本质都和地址计算有关。

例如：

  • 数组元素访问，本质上是“基地址 + 偏移量”
  • 指针运算，本质上是按类型大小做地址移动

所以这一章会进一步强化你对这句话的理解：

地址不是抽象符号，而是机器定位数据的真实线索。

3.12 结构体和异构数据

结构体和数组的区别在于：

  • 数组中的元素类型通常相同
  • 结构体中的字段类型可以不同

因此结构体在内存中，本质上是：

  • 一块连续区域
  • 各字段按固定偏移分布

编译器要做的事情，就是替你管理这些偏移关系。

所以从机器视角看：

  • 访问结构体字段，仍然是在做地址加偏移

3.13 本章最重要的思维模型

1. 变量名不是机器对象

机器不认识变量名，只认识寄存器、内存地址和常数。

2. 指令是在更新机器状态

程序执行不是抽象故事，而是状态一步步变化。

3. 控制结构最终都会落成跳转

if、while、for 的底层都离不开比较和跳转。

4. 函数调用需要独立上下文

调用函数不是简单跳转，而是要保存和恢复现场。

5. 栈适合处理嵌套调用

因为函数调用的返回顺序天然符合后进先出。

6. 指针和数组离不开地址运算

很多看起来“高级”的数据访问，底层只是按偏移找位置。

易混点整理

变量和存储位置

  • 变量名是源代码层概念
  • 机器真正操作的是寄存器和地址

可执行路径和代码结构

  • 代码里看到的是 if、while
  • 机器里真正发生的是比较、跳转和回跳

函数调用和普通跳转

  • 普通跳转只改控制流
  • 函数调用还要保存返回地址和运行现场

栈和队列

  • 栈是后进先出
  • 队列是先进先出
  • 函数调用适合用栈，不适合用队列

指针和值

  • 指针本身保存的是地址
  • 指针指向的位置里才是数据

学这一章时该怎么抓重点

如果你现在不想硬学汇编语法，那么第三章至少要保住这四件事：

  1. 知道寄存器、内存、立即数分别是什么
  2. 知道一条指令本质上是在改机器状态
  3. 知道 if、while 最后都会变成比较和跳转
  4. 知道函数调用为什么需要栈帧和保存现场

这样虽然还不能算把第三章吃透，但已经能把主线保住，不至于后面整本书断层。

一句话总结

第三章是在训练你从“源代码视角”切到“机器执行视角”：程序在底层并不是按变量名、语句块和函数名运行的，而是通过寄存器、内存地址、条件码、跳转和栈帧来一步步改变机器状态。

CS:APP 第二章

发表于2026-04-20|更新于2026-04-20|CS:APP
|浏览量:

本章主线

第二章的核心问题是：计算机究竟怎样用二进制位去表示整数、浮点数和地址，以及这些表示方式为什么会影响程序的计算结果。

这一章最重要的转变是：

  • 数学里的数是抽象对象。
  • 机器里的数是有限位模式。

也就是说，机器做的不是“纯数学运算”，而是“在固定位宽下对位模式做运算”。

2.1 信息存储

字节和地址

  • 大多数计算机使用 8 位为一个字节。
  • 内存可以看成一个很大的字节数组。
  • 每个字节都有唯一地址。

程序中的指针，本质上保存的就是某个内存位置的地址。

十六进制

  • 二进制太长，不方便看。
  • 十六进制正好每一位对应 4 个二进制位。
  • 所以在系统编程里，经常用十六进制表示地址、字节和机器码。

例如：

  • 0x2F 对应二进制 0010 1111

字节顺序

多字节对象在内存中的存放顺序有两种：

  • 大端法：最高有效字节放在低地址
  • 小端法：最低有效字节放在低地址

现在常见的 x86 机器一般是小端法。

一句话记忆：

  • 小端法更像“低位先放”
  • 大端法更像“高位先放”

2.2 整数表示

无符号数

  • 无符号数的每一位都参与表示数值大小。
  • 如果有 w 位，那么取值范围是：

0 ~ 2^w - 1

有符号数

CS:APP 里最核心的是补码表示。

对一个 w 位补码整数：

  • 最高位是符号位
  • 取值范围是：

-2^(w-1) ~ 2^(w-1) - 1

例如 8 位补码的范围是：

-128 ~ 127

补码为什么重要

补码的好处是：

  • 加法和减法可以统一到同一套硬件逻辑里
  • 正数和负数都能直接参加二进制加法

这也是现代计算机广泛使用补码的根本原因。

2.3 有符号数和无符号数之间的转换

这一块非常容易坑。

关键点不是“位变了”，而是：

位模式没变，解释方式变了。

例如一个 8 位模式：

11111111

  • 当成无符号数，是 255
  • 当成补码数，是 -1

所以这里再次呼应第一章那句话：

信息 = 位 + 上下文

2.4 扩展与截断

扩展

当一个较短位宽的整数转换成较长位宽时：

  • 无符号数做零扩展
  • 有符号补码数做符号扩展

例如：

  • 00001010 扩展到 16 位，会在前面补 0
  • 11110110 如果是负数补码，扩展时前面补 1

截断

当一个较长位宽的整数转换成较短位宽时：

  • 高位会被直接丢弃
  • 结果等价于对 2^k 取模，其中 k 是目标位宽

这也是很多“明明没写错却算错了”的根源。

2.5 整数运算

无符号加法

无符号数加法本质上是模 2^w 运算。

如果结果超过表示范围，就会回绕。

例如 8 位无符号数：

  • 255 + 1 = 0

补码加法

补码加法在位级上和无符号加法一样，区别只在结果如何解释。

如果结果超出补码表示范围，就会发生溢出。

乘法和移位

  • 左移通常相当于乘以 2^k
  • 右移要分逻辑右移和算术右移

对于有符号数：

  • 逻辑右移高位补 0
  • 算术右移高位补符号位

x86 上对有符号数右移，通常采用算术右移。

2.6 溢出

这一章必须牢牢记住：

机器整数运算不是无限精度的。

无符号溢出

  • 本质是回绕
  • 超过最大值后从 0 重新开始

有符号溢出

  • 结果超出补码可表示范围
  • 在 C 语言里，有符号溢出的行为要格外小心

常见现象：

  • 两个正数相加，结果变成负数
  • 两个负数相加，结果变成正数

这不是数学坏了，而是位宽不够了。

2.7 位运算

位运算是系统编程的基本功。

常见操作：

  • &：按位与
  • |：按位或
  • ^：按位异或
  • ~：按位取反

位运算常用来：

  • 取某几位
  • 设置某几位
  • 清除某几位
  • 判断奇偶
  • 实现掩码操作

这一部分的核心不是死背，而是能把十六进制、二进制和位模式联系起来。

2.8 浮点数

浮点数和整数是完全不同的一套表示方法。

一个浮点数通常可以拆成三部分：

  • 符号位
  • 阶码
  • 尾数

CS:APP 里主要基于 IEEE 754 标准理解浮点数。

浮点数最重要的几个结论

  • 浮点数不是精确实数
  • 很多十进制小数无法被二进制浮点数精确表示
  • 浮点数运算不一定满足结合律
  • 浮点数比较时要小心精度问题

例如：

  • 0.1 在二进制里通常不能被精确表示

所以程序中的浮点误差，不一定是程序员算错了，而可能是表示方式本身决定的。

第二章最重要的思维模型

1. 位模式和数值不是一回事

位模式是底层存储，数值是解释结果。

2. 位宽有限

所有整数运算都受位宽限制，因此会出现截断和溢出。

3. 同样的硬件加法，不同的解释方式

无符号加法和补码加法在位级上是同一套加法器，差别只在解释。

4. 转换经常比运算更危险

很多 bug 不是出在加法本身，而是出在有符号和无符号之间的隐式转换。

5. 浮点数是近似表示

不能拿实数世界的直觉，直接套到浮点数上。

易混点整理

无符号数和补码数

  • 位模式可能一样
  • 解释出来的数值可能完全不同

逻辑右移和算术右移

  • 逻辑右移补 0
  • 算术右移补符号位

零扩展和符号扩展

  • 无符号数补 0
  • 有符号数按符号位扩展

回绕和数学错误

  • 机器出现奇怪结果，很多时候不是算错
  • 而是固定位宽下发生了模运算或溢出

浮点误差和程序错误

  • 浮点误差不一定说明代码逻辑错了
  • 也可能只是浮点表示的天然限制

总结

第二章是在训练从“数学视角”切到“机器视角”：计算机里的整数和浮点数都不是无限精度的抽象对象，而是在固定位宽和特定编码规则下被表示、运算和解释的位模式。

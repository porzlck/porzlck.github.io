CS:APP 第一章

发表于2026-04-20|更新于2026-04-20|CS:APP
|浏览量:

本章主线

第一章的核心问题是：一个 hello.c 程序，怎样从文本文件一步步变成真正运行在机器上的进程，并最终把结果输出到屏幕。

1.1 信息就是位 + 上下文

  • 计算机中的所有信息，本质上都是一串二进制位。
  • 位本身只是材料，真正的含义取决于解释它的规则，也就是上下文。
  • 同一串位在不同上下文下，可以表示字符、整数、浮点数、机器指令或内存地址。

一句话记忆：位是材料，上下文决定含义。

1.2 程序被其他程序翻译成不同格式

一个 C 程序从源码到运行，大致经历下面几个阶段：

源代码 -> 预处理 -> 编译 -> 汇编 -> 链接 -> 可执行文件 -> 加载为进程 -> 运行输出

  • 预处理：展开 #include、#define 等预处理指令。
  • 编译：把 C 代码翻译成汇编代码。
  • 汇编：把汇编代码翻译成机器指令，生成目标文件。
  • 链接：把程序中引用到的库函数和其他目标文件接起来，生成最终的可执行文件。

易混点：

  • #include 主要提供函数声明，让编译器知道函数怎么用。
  • 链接阶段才去找到函数真正的定义和实现。

1.3 为什么要理解编译系统

理解编译系统很重要，因为它能帮助我们：

  • 理解程序为什么会出现编译错误或链接错误。
  • 更好地做性能优化。
  • 更清楚地理解恶意代码、缓冲区溢出等安全问题。

1.4 处理器如何执行程序

  • 可执行文件被操作系统加载到内存后，才会变成进程。
  • CPU 会不断执行“取指令、解释指令、执行指令”的循环。
  • 程序的运行，本质上就是机器状态在不断变化。

这里要分清两个概念：

  • 可执行文件：磁盘上的静态文件。
  • 进程：程序被加载到内存之后形成的运行实例，包含代码和运行时状态。

同一个可执行文件可以被多次运行，因此也可以对应多个不同的进程。

1.5 高速缓存至关重要

  • CPU 的速度远高于主存和磁盘。
  • 如果每次都直接从慢速存储设备取数据，程序会非常慢。
  • 因此系统引入缓存机制，把常用数据放在更快、更小的存储层里。

这里埋下了后面性能优化的核心思想：局部性。

1.6 存储设备形成层次结构

从快到慢、从小到大，大致可以看成：

寄存器 -> 高速缓存 -> 主存 -> 磁盘 -> 网络/远程存储

规律通常是：

  • 越快的存储越贵、越小。
  • 越慢的存储越便宜、越大。

整个系统通过层次结构，在成本和性能之间做平衡。

1.7 操作系统管理硬件

操作系统不会把底层硬件直接暴露给程序，而是提供了三个关键抽象：

进程

  • 让程序觉得自己独占 CPU。
  • 是对处理器的一种抽象。

虚拟内存

  • 让每个进程都觉得自己拥有一套连续、私有的地址空间。
  • 程序看到的是虚拟地址，不是裸物理地址。

文件

  • 把磁盘、终端、键盘、网络连接等 I/O 资源，尽量统一成可读写对象。
  • 本质上是操作系统对 I/O 设备的抽象。

1.8 网络也是系统的一部分

  • 网络可以看作一种特殊的 I/O 设备。
  • 一台机器不仅能执行本地程序，也能通过网络和其他系统通信。
  • 这为后面的网络编程、并发服务器等内容做铺垫。

1.9 本章中的几个重要主题

Amdahl 定律

  • 系统整体性能的提升，受限于最慢、最难优化的那部分。
  • 这提醒我们做优化时要找真正的瓶颈。

并发与并行

  • 并发：多个任务在时间上交错推进。
  • 并行：多个任务在同一时刻真正同时执行。

抽象

整本 CS:APP 都在反复强调抽象：

  • 文件是对 I/O 的抽象。
  • 虚拟内存是对内存系统的抽象。
  • 进程是对处理器执行的抽象。

抽象的作用不是把细节消灭，而是先把复杂性压住，让程序员能在合适的层上思考问题。

易混点整理

#include 和链接

  • #include 提供声明。
  • 链接提供实现。

可执行文件和进程

  • 可执行文件是静态文件。
  • 进程是运行中的程序，是“程序 + 运行时状态”。

物理内存和虚拟内存

  • 物理内存是真实硬件。
  • 虚拟内存是操作系统给进程提供的私有地址视图。

总结

第一章是在建立全局视角：程序从源码到运行，不只是“代码变机器指令”这么简单，而是一个涉及编译系统、处理器、存储层次、操作系统抽象和网络通信的完整计算机系统过程。
